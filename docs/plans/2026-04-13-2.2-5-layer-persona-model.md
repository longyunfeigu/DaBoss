## Context

Epic 2 需要把 Persona 从目前"markdown 文件 + 应用层 dataclass"升级为"结构化 5-layer 领域模型 + DB 持久化 + 证据链"。Story 2.2 是该转型的数据骨架 — 没有这个骨架，Story 2.3/2.4/2.7/2.8 都无法开工。

当前代码现状与 Story AC 的偏差（Phase 1/2 已澄清）：
1. `Persona` dataclass 目前定义在 **Application 层** (`application/services/stakeholder/persona_loader.py:22`)，不在 Domain 层 — AC1/AC2 要求迁入 Domain。
2. DB 目前**没有** `stakeholder_personas` 表（grep 确认）— AC4 实际是 CREATE TABLE，不是 ALTER。已与用户确认：新表同时包含 v1 + v2 字段。
3. 代码库**没有** `StakeholderRepository` 单一类，只有 `ChatRoomRepository/MessageRepository/...` 多个仓储 — AC6 实际是新建 `StakeholderPersonaRepository`。已与用户确认：按现有模式新建并注册到 UoW。

核心挑战：向后兼容 — 10+ 处调用 `PersonaLoader` 的代码不能回归；5 个现有 markdown persona 在 AC5 路径下必须继续可读。

- **这个 Story 能不能只做一半？** 不能。Persona 领域模型、表结构、仓储接口是一体的 — 只做 Domain dataclass 不建表的话，Story 2.3/2.4 没法消费。

## §4 风险

| 风险 | 等级 | 应对 |
|---|---|---|
| Persona 从 Application 移到 Domain 破坏 10+ 调用者的 import | 高 | 在 `persona_loader.py` 保留 `Persona` 类型别名：`from domain.stakeholder.persona_entity import Persona`；调用者不改 |
| JSONB 列在 SQLite 测试 DB 中不支持 | 中 | 使用 SQLAlchemy 的 `JSON` 类型（非 `JSONB`），跨 Postgres/SQLite 兼容；迁移里按 `sa.JSON()` 写 |
| `TEXT[]` (source_materials) 在 SQLite 不支持 | 中 | 用 `JSON` 列存 `list[str]`，避免方言差异 |
| v1/v2 混读导致 `list_personas()` 语义变化 | 中 | 约定：DB 中 v2 优先；v1 依赖文件系统保留兼容路径 |
| `get_with_evidence` 返回结构与 Evidence 位置分散设计不清 | 中 | Evidence 存在 Persona 的 `evidence_citations` 字段中（JSONB），不拆独立表 — Story 2.2 明确不新增 evidence 表 |
| Domain 实体导入 Infrastructure（DDD 违规） | 高 | blocking 检查：Domain 只能 `from __future__` / `dataclasses` / `typing` / `datetime` |
| Alembic auto-generate 漏写 JSONB 字段注释 | 低 | 手工编辑迁移脚本确认 `comment=` 完整 |

### 回滚方式

- 迁移支持 `alembic downgrade -1`
- 代码层面：撤销 PR；现有 markdown 加载路径保持独立，不受新代码影响

## §8 方案概述

### 架构草图

```
┌────────────────────────────────────────────────────────────────┐
│ Domain Layer                                                   │
│   persona_entity.py                                            │
│   ├── HardRule / IdentityProfile / ExpressionStyle /           │
│   │   DecisionPattern / InterpersonalStyle  (5 个 dataclass)   │
│   ├── Evidence  (dataclass + layer 校验)                       │
│   └── Persona  (扩展：5 层字段 + evidence_citations +          │
│                  schema_version + legacy_content + 保留 v1)    │
│                                                                │
│   repository.py                                                │
│   └── StakeholderPersonaRepository(ABC)                        │
│       ├── save_structured_persona(persona) -> Persona         │
│       ├── get_by_id(id) -> Optional[Persona]                  │
│       ├── get_with_evidence(id) -> tuple[Persona, list[Ev]]   │
│       └── list_all() -> list[Persona]                          │
└────────────────────────────────────────────────────────────────┘
              ▲                              ▲
              │                              │
┌─────────────┴──────────────┐   ┌───────────┴──────────────────┐
│ Infrastructure             │   │ Application                  │
│   StakeholderPersonaModel  │   │   PersonaLoader              │
│     (stakeholder_personas) │   │   ├── get_persona(id):       │
│   SQLAlchemyStakeholder    │   │   │   先查 DB, 找到 v2 →     │
│   PersonaRepository        │   │   │   返回; 否则扫 markdown  │
│   (_to_entity / _to_model) │   │   └── list_personas():       │
│                            │   │       DB v2 + MD v1 合并      │
└────────────────────────────┘   └──────────────────────────────┘
```

### 关键决策

1. **Persona 放 Domain，Application 层用 re-export 保持兼容**
   - `persona_loader.py` 顶部改为 `from domain.stakeholder.persona_entity import Persona`
   - 10+ 调用方 `from application.services.stakeholder.persona_loader import Persona` 继续有效
   - 未来 Story 可逐步改为直接从 Domain 导入

2. **Evidence 内嵌到 Persona.evidence_citations，不单独建表**
   - `get_with_evidence` 直接从 Persona JSON 字段返回 `tuple[Persona, list[Evidence]]`
   - 理由：Evidence 是 Persona 的强聚合成员，不是独立聚合；KISS

3. **TEXT[] 改 JSON 存 list[str]**
   - AC 原文写 `TEXT[]`，但测试 DB 是 SQLite，不支持数组
   - 用 `JSON` 列存 `list[str]`，Postgres/SQLite 都兼容
   - 语义一致，实现简单

4. **schema_version=1 的 Persona 永远从 markdown 读**
   - 本 Story 不向 DB 写任何 v1 数据
   - v1 persona 通过 `PersonaLoader._scan_markdown_dir()` 路径继续服务
   - v2 从 DB 加载；list 时两者合并

5. **Layer 校验放 Evidence 而不是枚举**
   - `Evidence.__post_init__` 检查 `self.layer in {"hard_rules","identity","expression","decision","interpersonal"}`
   - 不引入 enum，保持 dataclass 轻量
   - 用 `DomainValidationException` 抛出
